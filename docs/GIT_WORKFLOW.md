# GIT_WORKFLOW.md — Git 형상관리 규칙

## 브랜치 전략
```
master        ← 배포 가능한 안정 브랜치
└── feat/xxx  ← 기능 개발 브랜치 (작업 단위로 분기)
└── fix/xxx   ← 버그 수정 브랜치
└── chore/xxx ← 설정, 문서 등 기타
```

## 커밋 메시지 규칙
```
[날짜] type: 작업 내용

예시:
[2026-05-21] feat: 상품 목록 페이지 추가
[2026-05-21] fix: 장바구니 수량 버그 수정
[2026-05-21] style: ProductCard 레이아웃 수정
[2026-05-21] refactor: 상품 fetching 로직 분리
[2026-05-21] docs: README 업데이트
[2026-05-21] chore: 패키지 업데이트 / 설정 변경
```

## ⚠️ AI(Claude) Git 푸시 규칙
- AI는 절대 **사용자 승인 없이 push 하지 않는다**
- 작업 완료 후 반드시 아래 순서를 따른다:

```
1. 작업 완료
2. Claude가 변경 파일 목록 + 커밋 메시지 제안
3. 사용자 검토 및 승인 ("올려줘" / "확인" 등)
4. git add → git commit → git push 실행
```

## 작업 후 Claude 체크리스트
- [ ] 변경된 파일 목록 사용자에게 보고
- [ ] 커밋 메시지 초안 제안
- [ ] 사용자 승인 대기
- [ ] 승인 후 push 실행
- [ ] push 결과 보고

## 작업 로그
| 날짜 | 브랜치 | 작업 내용 | 커밋 |
|------|--------|----------|------|
